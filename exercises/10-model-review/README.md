# 演習 10: モデル選択と変更レビュー

**所要時間:** 25分  
**使う機能:** モデル選択、Diff レビュー、Git

## 目的

タスクに応じた**モデル選択**と、AI 出力を安全に取り込む**レビュー習慣**を身につける。

## 背景

モデルによって速度・精度・コストが異なります。また、AI の出力は必ずしも正しくありません。**信頼するが、検証する** が鉄則です。

## モデル選択の目安

| タスク | 推奨 | 理由 |
|--------|------|------|
| Tab 補完 | Cursor Tab（専用） | 低レイテンシ |
| Inline Edit | 高速モデル | 小さな変更 |
| コード理解 | 高速モデル | Ask は編集しない |
| 複数ファイル実装 | バランス型 | 精度と速度 |
| アーキテクチャ設計 | 高性能モデル | 深い推論 |
| 機械的リファクタ（大量） | 軽量モデル | コスト削減 |

Agent パネル上部のモデル選択から切り替えられます。

## 手順

### Step 1: 同じタスクを 2 モデルで比較（任意）

**タスク:**

```
@sample-app/src/app.js

todos 配列を id でソートして表示する機能を追加。昇順 / 降順切り替えボタンも。
```

1. 高速モデルで実行 → 結果をメモ（時間、品質）
2. 新しいチャットで高性能モデル → 比較

コストを気にするなら Step 2 以降だけで OK。

### Step 2: Diff レビューのチェックリスト

Agent の変更を Accept する前に、以下を確認:

```
□ 意図しないファイルが変更されていないか
□ 削除されたコードに必要なものが含まれていないか
□ ハードコードされた API キー / パスワードがないか
□ 既存機能（追加・削除・完了）が壊れていないか
□ スタイルがプロジェクトと一貫しているか
```

### Step 3: Git で変更を確認

```powershell
git status
git diff
git diff --stat
```

Agent に `@Commit`（Diff of Working State）を付けて:

```
@Commit

今の diff をレビューして。マージ前に直すべき点があれば指摘して。変更はまだしないで。
```

### Step 4: 動作確認

```powershell
start sample-app\src\index.html
```

手動テスト項目:

1. Todo 追加
2. 完了トグル
3. 削除
4. （演習04以降）フィルター
5. （今回）ソート

### Step 5: コミット（任意）

問題なければ:

```powershell
git add sample-app/
git commit -m "feat: add todo sort order toggle"
```

Agent にコミットメッセージ生成を任せても OK:

```
@Commit 適切なコミットメッセージを conventional commits 形式で 3 案提案して
```

## 成功条件

- [ ] モデル選択 UI を確認した
- [ ] Diff レビューチェックリストを 1 回以上実施
- [ ] `git diff` で変更全体を確認した
- [ ] ブラウザで動作確認した

## エスカレーションルール

同じバグで Agent が **2 回失敗** → モデルを上げる、または Ask で原因調査  
**3 回失敗** → 手動デバッグ、問題を小さく分割
